Retrieval Completeness Beats Memory Layers

A comparative field evaluation of six productivity-memory agent configurations across email, calendar, Slack, and cross-source fusion tasks.

Abstract

This report studies a 30-question memory benchmark run against a live productivity workspace. The benchmark tested whether agents could recover facts and synthesize state across Gmail-like email, calendar events, Slack messages, and cross-source links. Systems differed less in language-model reasoning than in retrieval access, pagination behavior, and connector reliability.

The highest observed score was Codex with code execution at 26/30, followed by Claude with Composio at 23/30, Micro native at 21/30, Hermes with Supermemory at 13/30, and OpenClaw with Composio CLI plus Hindsight at 12/30. A projected Micro variant with complete Slack coverage would score approximately 28/30.

Main finding Retrieval completeness dominated model sophistication. Memory layers underperformed when connectors returned shallow windows, missed calendar state, or failed under rate limits.

1. Evaluation Design

The benchmark consists of thirty questions grouped into seven dimensions. The tasks are intentionally operational: a correct answer requires reading live tool output, interpreting temporal relationships, and linking information across tools.

3. Analysis

3.1 Retrieval depth explains the ranking better than memory branding

The two memory-augmented stacks, Hermes plus Supermemory and OpenClaw plus Hindsight, did not place near the top. Hermes scored 13/30 after roughly 35 minutes with rate limits and errors; OpenClaw scored 12/30 after reasoning over a shallow retrieval window. The result does not imply that external memory is useless. It implies that memory cannot compensate for incomplete, stale, or flaky retrieval at answer time.

On this benchmark, the bottleneck was not remembering. It was reliably fetching the right slice of the user's workspace before reasoning.

3.2 Micro's actual gap was Slack access, not core reasoning

Micro's native stack scored 21/30 in less than five minutes while failing the Slack block because it lacked DM/private-channel coverage. The same adjudication projects 28/30 if Micro gets a clean 7/7 Slack block. That would place Micro above Codex while preserving the fastest runtime profile.

3.3 Codex won through exhaustive retrieval, with verification caveats

Codex benefited from code execution and deeper pagination, surfacing claims such as a 30,254-message unread count and an oldest unread email from November 2025. The transcript also flags two Codex claims as requiring caution because they conflicted with another native-tool recheck. Its lead is therefore best interpreted as a retrieval-depth advantage rather than a clean reasoning-only win.

3.4 Calendar errors are catastrophic for cross-source tasks

Hermes missed the same-day standup, inferred that there were no meetings that day, and then cascaded that error into meeting-prep, attention, and cross-source questions. Calendar is a small block numerically, but it anchors the temporal frame for the whole benchmark.

4. Failure Taxonomy

Observed failure modes

  • Pagination truncation: systems answered exact-count or oldest-item questions from the first page.
  • Connector shallowness: OpenClaw saw a tiny email window and treated it as the inbox.
  • DM/private-channel blindness: Micro and Hermes missed Slack DM context.
  • Temporal anchoring errors: Hermes selected a future PR review instead of the same-day standup.
  • Ambiguous negative detection: payment reminders, no-reply states, and bot mentions required careful interpretation.

Design implications

  • Expose total counts and pagination cursors as first-class model-visible evidence.
  • Separate public Slack search from DM/private-channel search in tool contracts.
  • Require date-window assertions before answering "today", "next", and "this week".
  • Report retrieval coverage with each answer, especially for exact-count tasks.
  • Prefer native domain APIs when they can provide complete state faster than generic connectors.

5. Discussion

The benchmark rewards complete state acquisition more than isolated semantic memory. A vector or personal memory layer can help recall prior facts, but the evaluated questions are mostly about current operational truth: unread counts, latest sent mail, upcoming meetings, Slack DMs, and cross-channel open loops. Those facts decay quickly and require authoritative connectors.

Native retrieval also matters for latency. Micro's sub-five-minute run suggests that domain-specific tools can provide a strong accuracy-speed tradeoff even when raw coverage is incomplete. Generic connector stacks can win when they retrieve deeply, as Codex did, but can fail badly when they retrieve shallowly, as OpenClaw did.

For production agents, the relevant metric is not only answer accuracy. It is answer accuracy per minute under rate limits, with explicit coverage reporting. A slower system that eventually paginates to ground truth may be valuable for audits; a faster native system may be better for everyday assistant workflows if it honestly reports missing scopes.

6. Conclusion

This field evaluation supports a practical conclusion: memory-agent quality is gated by retrieval completeness. The most capable systems were those that fetched enough live workspace state before synthesizing. The weakest systems were not necessarily weaker reasoners; they were operating on incomplete or misaligned context.

The strongest product direction is therefore straightforward: close Slack DM/private-channel access, enforce pagination-to-completion for exact queries, and make coverage evidence visible in every answer. Under that roadmap, the native Micro stack moves from a fast third-place result to the projected top of the table.

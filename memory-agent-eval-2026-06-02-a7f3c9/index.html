Memory Agent Evaluation Report

A text-first research note on a 30-question productivity-memory benchmark across email, calendar, Slack, and cross-source reasoning.

Evaluation date: June 2, 2026. Published as an unlisted GitHub Pages artifact with crawler indexing disabled.

Abstract

This report summarizes a live workspace benchmark comparing agent systems on operational memory tasks. The evaluation asked each system to answer 30 questions spanning email recall, email synthesis, calendar recall, calendar temporal reasoning, Slack recall, Slack synthesis, and cross-source fusion.

The main result is that retrieval completeness explains performance better than the presence of an external memory layer. Systems that paginated, accessed the relevant Slack surfaces, and anchored calendar time correctly outperformed systems that relied on shallow connector output or memory layers without complete current-state retrieval.

1. Executive Summary

Top observed system: Codex with code execution, scoring 26/30 in 22 minutes 20 seconds.

Fastest observed system: Micro native, scoring 21/30 in less than 5 minutes.

Best projected system: Micro native with complete Slack coverage, estimated at 28/30 in less than 5 minutes.

Core finding: memory layers did not compensate for incomplete retrieval. Hermes plus Supermemory and OpenClaw plus Hindsight scored near the bottom because they missed live calendar state, DMs, pagination depth, or complete inbox coverage.

2. Method

The benchmark contains 30 practical questions. Each question targets a concrete memory dimension and requires verifiable ground truth from live workspace data. The grading labels are:

  • Pass: answer is correct and supported by the available ground truth.
  • Partial: answer is directionally useful but incomplete, ambiguous, or dependent on an unexhausted retrieval window.
  • Fail: answer is incorrect, missing, or based on insufficient data treated as complete.

The score totals are the adjudicated totals from the source transcript. Some dimension summaries are shorthand diagnostic slices and should be read as qualitative breakdowns, not as independently recomputed totals.

5. System-by-System Notes

Codex: 26/30, 22m 20s

Codex produced the best observed score. Its advantage came from code execution and deeper pagination, especially on exact email-count and oldest-unread tasks. The main weakness was speed. The source transcript also flags two Codex claims for follow-up verification: the 30,254 unread count and the AWS top-sender claim, both of which conflicted with another native-tool recheck.

Claude: 23/30, 14m

Claude was the best balanced connector-based run. It performed well on Slack and cross-source reasoning and handled negative-detection questions carefully. It was slower than Micro and did not match Codex's exhaustive retrieval depth.

Micro actual: 21/30, less than 5m

Micro was the fastest observed system. The main gap was not general reasoning but missing Slack DM/private-channel access. Calendar and cross-source performance were strong in the source adjudication, and the system was explicit about retrieval limits such as pagination.

Micro projected: 28/30, less than 5m

The projection asks where Micro would rank if it scored 7/7 on Slack. Under that single change, Micro moves to 28/30, ahead of Codex, while retaining the fastest runtime profile. This supports the claim that the primary Micro bottleneck is a connector coverage gap.

Hermes + Supermemory: 13/30, roughly 35m

Hermes with Supermemory was rate-limited and difficult to run in one shot. It performed reasonably on some email questions, but missed the same-day standup and collapsed on calendar-dependent questions. This made it a clear example of a memory layer failing to compensate for unreliable current-state retrieval.

OpenClaw + Composio + Hindsight: 12/30

OpenClaw had broad theoretical connectors but retrieved too shallowly. It appeared to reason over a tiny email window, reported two unread emails where the real backlog was far larger, and missed several Slack and cross-source links. Its failure mode was not lack of reasoning but lack of data depth.

6. Main Findings

  1. Retrieval completeness is the dominant variable. The strongest systems fetched enough live workspace state before answering.
  2. Memory layers are not a substitute for current-state access. Supermemory and Hindsight did not help when the system missed meetings, DMs, or pagination.
  3. Slack access is decisive. DM/private-channel coverage changes the ranking substantially.
  4. Pagination is a first-class requirement. Exact counts, oldest items, and sender-frequency questions cannot be answered from one page.
  5. Latency matters. Codex scored highest observed accuracy but took more than four times as long as Micro.

7. Recommended Product Changes

  1. Add complete Slack DM/private-channel retrieval where permissions allow it.
  2. Expose count endpoints or page-to-completion helpers for exact email questions.
  3. Make retrieval coverage visible in every answer: pages read, total available, and whether `hasMore` remained true.
  4. Require explicit date-window anchoring before answering "today", "next", "earlier", or "this week".
  5. Separate bot mentions from human mentions in Slack tools.
  6. Prefer native domain APIs for fast, complete state retrieval over generic connector abstractions when the use case is operational memory.

8. Conclusion

The benchmark supports a simple conclusion: the best memory agents are not the ones with the fanciest memory layer; they are the ones that reliably retrieve complete, current, source-grounded state before reasoning. In this test, code execution won by paginating deeply, Claude performed well through broad connectors, Micro was fastest but Slack-capped, and the memory-augmented stacks underperformed because retrieval was incomplete or flaky.
